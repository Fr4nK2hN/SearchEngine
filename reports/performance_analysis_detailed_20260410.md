# 搜索引擎系统性能分析（详细版，2026-04-10）

## 1. 分析目的

本节旨在对当前搜索引擎系统的运行性能进行系统化分析，回答以下几个问题：

1. 当前系统在实际运行状态下的查询延迟处于什么水平？
2. 不同排序策略之间的性能差异有多大？
3. 系统当前的主要性能瓶颈位于哪个阶段？
4. 现有优化措施是否已经产生明显效果？
5. 当前系统是否已经达到“可演示、可答辩”的工程水平？

本分析尽量基于当前代码库和当前运行中的真实服务，不依赖主观推测。

### 2.3 当前部署方式的一个重要限制

当前 `app` 容器中的 Gunicorn 启动命令为：

```bash
gunicorn --no-control-socket --bind 0.0.0.0:5000 app:app
```

该命令没有显式指定 `--workers`，因此当前部署等价于 **单 worker、同步处理模型**。

这意味着：

- 本次测试结果主要反映的是**单请求延迟**
- 不代表系统在高并发场景下的吞吐能力
- 当某个 `cross_encoder` 请求较慢时，其他请求可能被阻塞

因此，本分析结论适合用于说明“单次查询响应性能”，但不能直接等价为“高并发性能评估”。

## 4. 性能指标定义

为了使报告中的结果更容易解释，本次分析使用以下指标：

### 4.1 总延迟指标

- `平均总耗时（Avg Total Latency）`
  - 表示单次查询从进入后端到生成结果完成的平均耗时
- `P50`
  - 中位数延迟，反映常规情况下的典型体验
- `P90 / P95 / P99`
  - 高分位延迟，反映尾部慢请求情况

### 4.2 分阶段延迟指标

- `retrieval_ms`
  - Elasticsearch 检索阶段耗时
- `feature_ms`
  - 特征提取阶段耗时，主要用于 LTR
- `inference_ms`
  - 模型推理耗时，主要对应 Cross-Encoder 或 Hybrid 重排序

### 4.3 embedding 相关指标

- `candidate_docs`
  - 参与排序阶段的候选文档数
- `precomputed_content_emb_count`
  - 候选文档中已具备正文 embedding 的数量
- `precomputed_title_emb_count`
  - 候选文档中已具备标题 embedding 的数量

这些指标用于判断当前系统是否仍在发生在线 embedding 补算。

## 7. 结果解读

### 7.1 Baseline：当前最快路径

`baseline` 平均总耗时仅 `17.67 ms`，且几乎全部耗时都来自 Elasticsearch 检索阶段。

特点：

- 路径最短
- 没有特征提取
- 没有深度模型推理

这表明当前 Elasticsearch 检索本身已经比较轻量，系统的基础召回速度是足够快的。

### 7.2 LTR：已经非常接近 baseline

`ltr` 平均总耗时为 `20.07 ms`，仅比 baseline 高约 `2.40 ms`。

它的分阶段开销为：

- 检索：`11.68 ms`
- 特征：`7.54 ms`
- 推理：`0.81 ms`

这说明：

1. 之前最重的 LTR 路径已经被优化到了非常可接受的水平
2. LTR 当前不再是系统瓶颈
3. embedding 预计算确实显著降低了在线排序成本

从工程角度看，`LTR` 已经从“较重排序模块”变成了“可常态在线运行的轻量重排模块”。

### 7.3 Adaptive：当前默认策略的合理性

`adaptive` 平均总耗时为 `43.18 ms`，明显慢于 `baseline` 和 `ltr`，但远快于 `hybrid` 和 `cross_encoder`。

这说明当前默认 adaptive 的核心价值不在于“最快”，而在于：

- 在轻查询上尽量省时
- 在难查询上启用更强排序
- 在效果和延迟之间取得可接受折中

从这个结果看，当前 adaptive 确实实现了这一目标。

### 7.4 Hybrid：比 Cross-Encoder 轻很多，但仍然不轻

`hybrid` 平均 `92.20 ms`，比 `cross_encoder` 快很多，但仍明显高于 `adaptive`。

其主要原因是：

- 虽然先做了 LTR 过滤
- 但仍需要对 top-k 结果做 Cross-Encoder 推理
- 因而推理成本依旧明显

这使得 `hybrid` 更适合作为实验策略或高质量备选，而不一定适合当前默认在线使用。

### 7.5 Cross-Encoder：当前最明显的性能瓶颈

`cross_encoder` 平均总耗时 `337.03 ms`，P95 达到 `502.58 ms`。

其耗时几乎全部来自推理阶段：

- 平均推理：`324.24 ms`

这一结果说明：

- 当前系统最大的单点瓶颈不是 ES 检索
- 也不是 LTR
- 而是 Cross-Encoder 推理本身

## 9. Adaptive 路由行为分析

本轮 `adaptive` 共处理 `12` 条查询，路由分布如下：

- `hard -> cross_encoder`: `5`
- `easy -> baseline`: `4`
- `easy -> ltr`: `3`

换算比例：

- hard 查询：`41.67%`
- easy baseline：`33.33%`
- easy ltr：`25.00%`

这说明当前 adaptive 并不是“固定走同一种策略”，而是在做真实分流。

同时，由于 hard 路径比例接近 `40%`，所以 adaptive 的平均耗时会被明显抬高。  
如果未来需要继续降低 adaptive 延迟，最直接的方向不是继续压 LTR，而是：

- 降低 hard 比例
- 或进一步压缩 hard 路径中的推理成本

因此当前结果可以视为“正常查询场景下的在线性能表现”，而不是最坏情况上界。

## 13. 当前性能瓶颈总结

### 13.1 已基本解决的问题

以下性能问题已经被明显改善：

- 文档 embedding 在线补算问题
- LTR 特征提取过慢问题

证据：

- embedding 覆盖率达到 `100%`
- `LTR` 平均总耗时已降至 `20.07 ms`

### 13.2 当前最主要瓶颈

当前最主要瓶颈是：

- `Cross-Encoder` 推理

证据：

- `cross_encoder` 平均推理 `324.24 ms`
- `hybrid` 平均推理 `77.36 ms`
- `adaptive` 的额外成本也主要来自其 hard 路径推理

### 13.3 当前部署层面的潜在问题

虽然单次查询延迟已经较好，但当前部署仍有两个结构性风险：

1. **Gunicorn 单 worker**
   - 在并发请求下，重查询可能阻塞其他请求
2. **Elasticsearch 占用内存较高**
   - 当前内存使用约 `4.45 GiB`
   - 对部署机器内存提出较高要求

## 14. 综合结论

基于当前代码库和 `2026-04-10` 的在线实测，可以得出以下结论：

### 14.1 当前系统已经达到可演示水平

从单次查询响应时间看：

- `baseline` 和 `ltr` 已经处于非常可用的水平
- `adaptive` 在质量与延迟之间给出了合理折中

因此，从毕业设计演示与答辩角度，当前系统已经具备较好的工程完成度。

### 14.2 当前默认策略选择是合理的

当前默认的 `adaptive` 配置虽然不是最快，但它能够：

- 避免所有查询都走最重模型
- 保持明显优于全量 `cross_encoder` 的在线速度
- 在简单查询上保留轻路径优势

因此，它作为默认在线策略是合理的。

### 14.3 性能优化的重点已经发生转移

系统早期的主要问题是：

- embedding 在线计算
- LTR 特征抽取过重

而现在，主要瓶颈已经转移到：

- `Cross-Encoder` 推理本身

这说明前期优化方向是有效的，系统已经进入新的优化阶段。

## 15. 后续优化建议

### 15.1 第一优先级：降低 hard 路径成本

建议优先从以下方向继续优化：

- 进一步降低 `hard -> cross_encoder` 的触发比例
- 继续压缩 hard 路径 `top-k`
- 尝试更轻量的 Cross-Encoder
- 增加更精细的 query difficulty 判别

### 15.2 第二优先级：提升并发能力

当前系统的单次延迟表现已经较好，但并发能力仍偏弱。建议：

- 为 Gunicorn 显式配置多个 worker
- 在报告中将当前测试明确标注为“单请求延迟分析”
- 后续补做并发压测

### 15.3 第三优先级：优化 Elasticsearch 维护状态

建议后续关注：

- 已删除文档比例偏高的问题
- segment merge 或重建索引带来的存储整理效果
- ES 内存配置与 JVM 参数优化
