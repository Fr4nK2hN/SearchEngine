## 第1页：方法定义（你做了什么创新）

### 1. 研究问题
传统方案对所有查询使用同一个重排器，导致：
- 简单查询过度计算（浪费延迟）
- 困难查询重排不足（损失质量）

### 2. 我的方法
提出 `Query Complexity / Uncertainty-Aware Adaptive Reranking`：

- 输入：查询特征（长度、词项数、停用词比例、prefix-like 等）
- 路由器输出：`P(hard | q)`
- 决策：
  - easy query -> `LTR`
  - hard query -> `Hybrid`
  - hard query 内再按 `delta = P(hard|q)-threshold` 选择重排深度 top-k

最终策略：
- `threshold=0.45`
- `hard_topk_policy=0.08:30,0.10:20,1.00:30`

### 3. 一句话贡献
我不是只换了一个 reranker，而是把多个 reranker 作为“专家池”，通过轻量 router 做动态策略选择（模型选择 + 重排深度选择）。

## 第2页：主结果与消融（证据链）

### 1. 主结果（5 seeds, 每 seed 60 queries）

| Method | nDCG@10 | Avg Latency (ms) | P95 (ms) |
| --- | --- | --- | --- |
| Baseline | 0.7546±0.0036 | 0.00±0.00 | 0.00±0.00 |
| LTR | 0.7914±0.0044 | 266.54±10.37 | 573.71±24.07 |
| Cross-Encoder | 0.9124±0.0041 | 230.59±1.29 | 279.36±1.92 |
| Hybrid | 0.8127±0.0037 | 343.40±10.23 | 646.05±21.83 |
| Adaptive | 0.8576±0.0024 | 369.77±12.94 | 713.19±28.09 |

关键读法：
- `Adaptive` 相对 `Baseline` 的 nDCG 绝对提升 `+0.1030`，相对提升约 `+13.66%`。
- 路由硬查询比例约 `45.33%`，说明不是“全量走重模型”。

### 2. 消融结果（创新点是否有效）

| Method | nDCG@10 | Avg Latency (ms) |
| --- | --- | --- |
| Fixed-Hybrid@30 (No Routing) | 0.9018±0.0027 | 497.13±10.00 |
| Router + Fixed Top-K | 0.8576±0.0024 | 369.77±12.94 |
| Router + Adaptive Top-K | 0.8576±0.0024 | 368.75±13.27 |

关键结论：
- `Router + Adaptive Top-K` 对比 `Router + Fixed Top-K`：
  - nDCG 不下降（`Δ=0.0000`）
  - 平均延迟小幅下降（`Δ=-1.02ms`）
- 这证明“自适应 top-k”至少在当前数据上是**质量不劣 + 延迟更优**的有效改动。

## 第3页：结论、局限与答辩问答

### 1. 最终结论
- 方法层面：完成了“路由 + 自适应重排深度”的方法设计。
- 结果层面：给出了主表和消融证据，能证明创新模块有效。
- 工程层面：Docker 可复现，参数和策略已落地到在线接口。

### 2. 已知局限（主动说，反而加分）
- 当前数据集 `first_doc_is_max_ratio` 较高，导致 MRR/Recall 区分度不足，nDCG 更有解释力。
- 样本规模仍偏小，后续可增加真实反馈样本并做显著性检验。

### 3. 老师常问问题（建议回答）
- Q: 你这个是不是只是调参？
  - A: 不是。核心是方法结构变化：从单模型重排变成“专家池 + 路由 + 深度自适应”，参数只是该结构下的实例化。
- Q: 为什么不直接全用最强模型？
  - A: 全用强模型延迟成本过高；本方法目标是质量-效率折中，通过 query-aware 决策减少不必要重排。
- Q: 你的创新到底在哪？
  - A: 创新点是“查询复杂度/不确定性感知的动态策略学习”，包括模型选择与重排深度联动控制。

### 4. 现场收尾一句话
本工作把“固定重排”升级为“动态重排控制”，在当前实验中实现了可复现的质量提升，并通过消融验证了策略设计的有效性。
